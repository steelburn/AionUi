# Aion App Framework — 产品方案

> 版本：v0.1 | 日期：2026-04-02 | 作者：产品-郭聪明

---

## 1. 产品定位与用户价值

### 1.1 不是 App Store，是活的 AI 应用生态

传统 App Store 分发的是静态软件包。Aion App Framework 分发的是**会思考的应用**。

每个应用 = 代码逻辑 + N 个 AI Agent。应用内的 AI 不是装饰品，是真正的执行主体：它理解用户意图、在运行时做决策、和其他应用的 AI 协商接口、按需改造自己的行为。

**核心用户价值（按重要性排序）：**

1. **自然语言即开发** — 用户说"做个读 GitHub PR 评论然后自动回复的应用"，AI 按框架规范生成代码，不合规则拒绝安装。开发门槛从会写代码降低到会描述需求。
2. **应用间 AI 动态协商** — 不是静态 API 对接，两个应用的 AI 可以互相发现、协商接口协议、甚至改造自己适配对方。这是和所有现有方案的本质区别。
3. **高级权限开放** — CLI 执行、Git 操作、截屏、录音等系统级能力，通过框架的权限声明机制安全开放给应用。
4. **本地优先** — 所有应用默认本地运行，数据不离本机。发行是可选行为，不是强制。

### 1.2 目标用户

- **主要用户**：重度使用 AI 工具的开发者 / 独立创作者，有"想做个小工具"的念头但不想写一整个项目
- **次要用户**：已有 Aion 扩展开发经验的人，想把自己的扩展升级为"有 AI 大脑"的应用
- **不是目标**：企业级软件采购、纯非技术用户（MVP 阶段）

## 2. 核心概念定义

用普通人能懂的语言解释五个核心概念：

### 2.1 App（应用）

一个 Aion App 是一个**打包好的小程序**，里面既有代码（做什么），也有一个或多个 AI（怎么决策）。

和普通扩展（Extension）的区别：普通扩展是给 Aion 加功能按钮；App 是一个独立运行的程序，有自己的 AI 大脑，能和其他 App 说话。

技术上：App = 符合框架规范的 Extension manifest + 至少一个 Agent 配置 + 可选的 UI 入口。

### 2.2 Protocol（通信协议）

两个 App 的 AI 想合作，需要说同一种语言。Protocol 就是这个语言规范。

关键设计：**Protocol 不是提前写死的**。App A 的 AI 发现 App B 后，双方 AI 可以协商："你能处理什么输入，我能提供什么格式，我们用这个协议对接吧"。协商结果缓存下来，下次直接用。

这是和 MCP、REST API 最大的区别：MCP 的工具是静态注册的；Aion Protocol 是 AI 在运行时动态协商的。

### 2.3 Runtime Engine（运行时引擎）

应用运行的地基。它负责：
- 给 App 分配资源（内存、进程沙箱）
- 管理权限（App 申请了什么权限，用户批没批）
- 维持应用间的通信通道
- 监控 App 是否在按规矩做事

用户感知不到 Engine，就像你感知不到操作系统的调度器一样。但没有它，所有 App 都乱套。

### 2.4 Mailbox（信箱）

类似 Claude 的 Team Mode：每个 App 有一个信箱地址（`app-name@aion`）。

App 之间通过信箱发送消息。消息不是实时调用，是异步投递。好处是：
- 发送方不需要知道接收方现在在不在线
- 接收方可以选择接受或忽略
- 消息有完整记录，可以追溯

信箱是 App 发现彼此的机制：所有已安装的 App 都在信箱目录里，你的 AI 可以查目录找到能帮你的 App。

### 2.5 Framework（框架）

App 开发者必须遵守的规范集合。包括：
- **项目结构规范**：文件怎么放、manifest 怎么写、Agent 配置放哪里
- **权限声明规范**：用了什么系统能力必须提前声明
- **通信接口规范**：怎么注册信箱、怎么发消息、怎么响应协商请求
- **Agent 协调规范**：多个 AI 之间怎么分工、怎么传递上下文

框架不仅是规范文档，还有校验工具。AI 生成的代码在安装前会跑校验，不达标不让装。

## 3. 核心场景

### 场景一：git-review-app + slack-notify-app 联动 Code Review

**用户描述**：我想让人提 PR 的时候，自动 review 代码，然后把结果发到 Slack 频道。

**发生了什么**：

1. `git-review-app` 的 Agent 监听 Git 仓库，检测到新 PR
2. Agent 执行代码分析（使用框架提供的 Git 高级权限接口：`framework.git.getPrDiff(prId)`）
3. Agent 生成 review 意见后，查询信箱目录，发现已安装 `slack-notify-app`
4. 两个 App 的 AI 协商接口：

   ```
   git-review-app: 我有一条结构化的 review 结果，格式是 {summary, issues[], severity}
   slack-notify-app: 我可以接受 {message: string} 或 {blocks: SlackBlock[]}
   git-review-app AI: 我来把我的格式转成你的 blocks 格式
   ```

5. 协商完成，`git-review-app` 把 review 结果转换后投递到 `slack-notify-app` 的信箱
6. `slack-notify-app` 收到消息，发到指定 Slack 频道

**用户的感知**：两个 App 装上去之后，自动就联动了，没有配置任何 API 对接。

**验收标准**：
- 从 PR 创建到 Slack 收到消息，全程无人工干预
- 如果 `slack-notify-app` 未安装，`git-review-app` 记录日志但不报错崩溃

---

### 场景二：voice-input-app 对接 meeting-notes-app

**核心挑战**：语音 App 输出的是音频流，笔记 App 期望的是文字。这不是格式问题，是接口不兼容。

**AI 动态改造自身的关键案例**：

1. `voice-input-app` 的 AI 发现 `meeting-notes-app` 想要接收输入
2. 协商发现接口不兼容：语音 vs 文本
3. `voice-input-app` 的 AI 判断：**我可以在输出前加一步 transcription**
4. AI 生成一段适配代码（满足框架规范），在运行时注入自己的 output pipeline
5. 现在 `voice-input-app` 对外输出变成了文字，`meeting-notes-app` 可以正常接收

这就是"应用改造自己适配对方"。不是人工写代码，是 AI 在运行时动态完成的。

**验收标准**：
- 无需用户手动配置转写步骤
- 改造行为在 App 日志里有完整记录
- 下次连接时直接复用协商结果，不重复改造

### 场景三：自然语言开发一个新应用

**用户对话**：

> 用户：帮我做一个 App，每天早上 9 点读我的 GitHub Notifications，过滤掉机器人的评论，剩下的发到我的邮箱，按仓库分组。

**发生了什么**：

1. Aion 的开发 Agent 解析用户意图，拆解为：
   - 定时触发（9:00）
   - GitHub API 调用（Notifications）
   - 过滤逻辑（排除 bot）
   - 邮件发送
   - 分组聚合

2. Agent 查询框架规范，确定需要申请权限：
   - `network: { allowedDomains: ["api.github.com"] }`
   - `network: { allowedDomains: ["smtp.*"] }`
   - `shell: false`（不需要）

3. Agent 生成完整的 App 包（manifest.json + 代码 + Agent 配置）

4. 框架校验工具运行校验：
   - manifest 格式是否合规？✓
   - 权限声明是否和代码实际使用一致？✓
   - Agent 配置是否有效？✓

5. 用户看到校验结果，确认安装

6. App 安装，定时任务注册，明天 9:00 自动触发

**验收标准**：
- 从用户描述到安装完成，全程不超过 3 分钟
- 如果生成的代码权限和 manifest 声明不一致，校验必须拦截，不能装上去
- 用户可以在安装前查看 App 申请了什么权限

## 4. 与 MCP / Skills 的区别和层级关系

```
┌─────────────────────────────────────────────────────────┐
│                  Aion App Framework                     │
│  完整的应用单元：代码 + AI + 生命周期 + 应用间通信         │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Aion Extension (现有)               │   │
│  │  静态功能扩展：UI组件、Adapter、主题、Settings Tab  │   │
│  │                                                 │   │
│  │  ┌──────────────┐  ┌─────────────────────────┐ │   │
│  │  │  MCP Server  │  │  Skills (.claude/skills) │ │   │
│  │  │  工具/数据源  │  │  Agent 执行规范          │ │   │
│  │  └──────────────┘  └─────────────────────────┘ │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

| 维度 | MCP | Skills | Extension | App（本方案）|
|------|-----|--------|-----------|------------|
| 本质 | 工具/数据接口 | Agent 行为规范 | 静态功能包 | 有 AI 的活程序 |
| AI 决策 | 无（被动调用） | 是执行者 | 无 | App 内置 AI |
| 应用间通信 | 无 | 无 | EventBus（同进程） | 信箱（跨App异步）|
| 接口协商 | 静态注册 | 无 | 无 | AI 动态协商 |
| 生命周期 | 无 | 无 | install/activate | install/run/pause/uninstall |
| 开发方式 | 手工编码 | 手工编写规范文档 | 手工编码 | 自然语言生成 |

**明确的层级关系**：App Framework 构建在现有 Extension 系统之上，不替代它。一个 App 在技术上仍然是一个特殊的 Extension，只是多了 AI 配置和通信协议能力。MCP Server 仍然是 App 可以使用的工具来源之一。

## 5. 应用生命周期

```
自然语言描述
     │
     ▼
AI 开发阶段（由开发 Agent 完成）
  - 拆解需求
  - 生成代码、manifest、Agent 配置
  - 自动申报所需权限
     │
     ▼
框架校验（本地 + 强制通过）
  - manifest schema 校验
  - 权限声明与代码使用一致性校验
  - Agent 配置有效性校验
  - 不通过 → 拒绝，返回错误报告给开发 Agent
     │
     ▼
用户审核（轻量）
  - 展示权限申请摘要
  - 用户确认或取消
     │
     ▼
本地安装
  - 注册信箱地址
  - 激活生命周期（onInstall → onActivate）
  - 定时任务、事件监听等注册
     │
     ▼
运行中
  - 响应触发条件（用户指令 / 定时 / 其他App信箱消息）
  - AI 处理逻辑 → 执行动作
  - 可暂停 / 恢复
     │
     ▼
可选：发行到 Aion App 目录（未来）
  - 作者提交 + 基本自动审核
  - 其他用户搜索安装
```

## 6. MVP 范围建议

### MVP 目标
让一个技术用户能用自然语言描述需求，生成一个符合规范的本地 App，并让两个 App 通过信箱完成一次数据传递。

### MVP 包含（P0）

| 功能 | 说明 |
|------|------|
| 框架规范文档 | manifest 格式、项目结构、权限声明规范，机器可读 |
| 框架校验工具 | 安装前自动跑，不通过拒绝安装 |
| 开发 Agent | 接受自然语言，生成符合规范的 App 包 |
| 信箱基础实现 | 注册地址、发送消息、接收消息、目录查询 |
| 高级权限接口（基础集）| CLI 执行、文件读写（指定目录）、HTTP 请求 |
| App 生命周期管理 | install / activate / deactivate / uninstall |
| 本地安装流程 | 权限摘要展示 → 用户确认 → 安装 |

### MVP 不包含（P1/P2）

| 功能 | 理由 |
|------|------|
| AI 动态协商（自动改造自身）| 技术复杂度高，用手工声明接口格式先跑通流程 |
| App 发行目录（线上） | MVP 先做本地，发行是下一步 |
| 截屏 / 录音权限 | 安全审计成本高，先做网络和文件 |
| 多 Agent 协调框架（复杂版）| MVP 每个 App 先允许 1 个主 Agent |
| 依赖 App（App 之间声明依赖）| 先用运行时信箱发现，不做安装时依赖解析 |

### MVP 验收标准（端到端）

1. 用户描述"做个每日读 GitHub Notifications 发邮件的 App"
2. 开发 Agent 在 5 分钟内生成 App 包
3. 框架校验通过
4. 用户看到权限摘要，确认安装
5. App 运行，行为符合描述
6. 手动触发另一个已安装的测试 App 向该 App 信箱发消息，该 App 正确响应

## 7. 规范不达标的处理机制

### 7.1 拦截时机

规范校验发生在安装前，**本地安装也不例外**。没有"先装上去再说"的选项。

### 7.2 拦截后的处理

```
校验失败
   │
   ├── 如果是 AI 生成的 App
   │      → 错误报告回传给开发 Agent
   │      → Agent 自动修复，重新提交校验
   │      → 修复失败超过 3 次 → 告知用户，要求澄清需求
   │
   └── 如果是用户手动导入的 App
          → 展示详细错误列表（哪一项不达标，规范是什么）
          → 用户可以选择：修改后重新提交 | 放弃安装
```

### 7.3 错误类型分级

| 级别 | 例子 | 处理 |
|------|------|------|
| 致命（必须修复）| manifest 缺少必填字段、权限声明和代码不一致 | 硬拦截，无法安装 |
| 警告（建议修复）| 缺少 description、apiVersion 未声明 | 提示用户，但允许安装 |
| 信息（仅提示）| 权限申请了但代码未使用 | 日志记录，不阻断 |

### 7.4 不能放行的情况

以下情况**无论如何不允许安装**，没有覆盖选项：
- 代码实际使用了未在 manifest 中声明的权限（如访问网络但未声明 network: true）
- Agent 配置指向不存在的 contextFile
- manifest 使用了保留前缀（aion-、system- 等）

## 8. 风险与歧义

### 8.1 AI 动态协商的边界（需要架构组明确）

"应用间 AI 动态协商，甚至改造自己适配对方"——这句话在 MVP 阶段实现到什么程度？

目前产品侧的判断：MVP 先实现手工声明接口格式（静态协商），AI 自动改造自身放到 P1。但这个边界需要架构组确认技术可行性和成本，才能最终定。

### 8.2 "自然语言开发"的失败处理

AI 生成的代码可能：不符合框架规范、权限声明有误、逻辑和用户描述有偏差。

MVP 阶段对用户的承诺应该是"帮你生成初稿"，不是"完全自动完成"。用户需要有检查和修改的能力。这个预期需要在 UI/UX 上明确传达，不能让用户以为描述完就万事大吉。

### 8.3 安全边界：谁来校验校验器

框架校验工具本身也是代码。如果校验逻辑有漏洞（比如权限检查绕过），App 可能在本地做不该做的事。这个问题 MVP 阶段先靠"校验工具由 Aion 团队维护，不允许第三方覆盖"来兜底，但长期需要专项设计。

---

## 9. 与老乔的待对齐点

本方案初稿由郭聪明单独完成，以下几个判断需要和老乔复核：

1. **MVP 是否包含 AI 动态协商**：郭聪明判断 MVP 不包含（风险太高），老乔可能有不同看法
2. **信箱是否应该支持同步请求**：方案里只有异步投递，同步 RPC 模式是否需要在 MVP 里覆盖
3. **开发 Agent 的能力边界**：它是否需要运行代码做验证，还是只做静态生成
4. **权限警告级别的设计**：有些权限（如 shell: true）用警告是否足够，还是应该升为致命错误

## 产品-技术对齐意见

> 对照技术方案（docs/proposals/aion-app-store-tech.md）后补充，2026-04-02

### Section 9 四个待决策项的产品判断

**1. 动态协商是否需要人工确认节点？**

产品判断：**P0 不需要，P1 需要**。

理由：MVP 阶段动态协商压根没有（技术方案 P0 不包含），等到 P1 实现协商时，再设计确认节点。到时候的产品设计建议是：首次协商完成后展示"这两个 App 达成了什么协议"给用户看，用户可以批准或拒绝，缓存后不再重复弹窗。一次性确认即可，不要每次通信都审批。

**2. 协商失败的降级策略？**

产品判断：**超时后回退静态 schema，而不是直接断开。**

理由：直接断开意味着用户的工作流中断，代价太高。回退静态 schema 相当于"用保守模式先跑通"，用户感知是"部分功能可能不完整"而不是"彻底失败"。但要在 UI 上明确告知："协商未能完成，已使用基础兼容模式连接"，不能静默降级。

超时时间建议 30 秒，超时次数达到 3 次后标记该连接为"需要人工介入"，停止自动重试。

**3. 本地 Store 的 App 来源？**

产品判断：**P0 仅支持用户手动安装，不做内置目录。**

理由：P0 还没有验证过 App 质量标准，内置目录需要有审核和维护成本。先让用户从文件安装，跑通一个完整的生命周期，再考虑目录。内置目录和远程 Store 一起放到 P1。

**4. Worker Thread vs Child Process？**

产品判断：**V0 Worker Thread 可接受，但需要在 UI 上设置资源预期。**

这是技术约束，产品能做的是：在 App 安装和运行界面说明"当前版本使用轻量沙箱，多个 App 同时运行时性能可能受影响"，避免用户误以为隔离是完整的。用户实际感知的风险：同时运行 3 个以上 App 时主进程卡顿。这个阈值需要架构组给出基准数据后再标注。

---

### 技术方案新增的产品约束（产品方案原来没覆盖的）

**约束1：权限 L2 运行时执行目前是空的**

技术方案 Section 6.1 指出：现有 `permissions.ts` 的运行时 enforcement 尚未实现，标注为 P2。但产品方案在 Section 7 里明确说了"代码实际使用了未在 manifest 中声明的权限 → 无论如何不允许安装"。

**这里有矛盾**：静态安装时能拦截，但运行时如果 App 通过动态方式绕过权限声明，当前架构拦不住。

产品修正：Section 7.4 的"无论如何不允许"要加注释——"安装时通过静态分析拦截；运行时执行层面的 enforcement 在 P1 补齐（对应技术方案的权限 L2）"。在 L2 未上线前，产品对外的承诺是"安装前静态校验"，不是"运行时100%拦截"，两者需要分开表述，不能混为一谈。

**约束2：每个 App 的 Agent 并发上限是 5 个**

技术方案 Section 6.3 里设定了默认上限 5 个。产品方案没有提过这个限制，场景描述里也没有触碰边界。

产品需要回答：这个 5 的上限是否需要暴露给用户，还是纯内部限制？

产品判断：暴露给用户，但以"当前活跃 Agent 数"来呈现，不说"上限 5 个"（避免被解读为功能弱）。用户看到的信息是"该 App 正在运行 3 个 AI"，超过上限时提示"已达到当前版本并发限制，请等待现有 Agent 完成后再触发新任务"。

**约束3：信箱发送速率限制 100 msg/s**

技术方案 Section 6.3 设了速率限制（100 msg/s/App）。产品场景一（git-review + slack-notify）和这个上限没有冲突，但如果未来有高频消息场景（如实时语音流分片发送），这个限制会是产品瓶颈。

当前 MVP 场景内没问题，标记为未来需要关注的设计边界。

---

### 产品方案遗漏、技术方案有、需要补的内容

**遗漏1：App 的"暂停/恢复"状态如何对用户呈现**

技术方案 Section 4.2 有 AppRuntime 的 start/stop 方法，但没有 pause/resume。产品方案 Section 5 的生命周期图里写了"可暂停/恢复"，但技术方案实际没有这个状态。

需要对齐：是产品砍掉"暂停"这个概念，还是技术方案补上 pause/resume 接口？

产品建议：暂停 = stop（不保留内存状态），恢复 = restart（重新加载），产品层面统一说"暂停"，底层是 stop+restart 而不是 suspend。这样技术不需要额外实现，用户也有直觉相符的操作。

**遗漏2：AI 开发 Agent 的身份没有定义**

产品方案 Section 3 场景三提到"开发 Agent 接受自然语言，生成 App 包"，但没有说这个 Agent 是什么、从哪来、靠什么 model 运行。技术方案 Section 7 同样只描述了流程，没有说开发 Agent 的配置。

产品需要补一个定义：开发 Agent = Aion 内置的一个特殊 Assistant，默认使用最强可用的模型，contextFile 包含完整的 App Framework 规范文档。它不是用户自己的 App，它是 Aion 的原生能力。

这个定义需要在产品方案里补清楚，否则开发组不知道边界在哪。

---

### 两份方案总体对齐情况

整体对齐度高，核心分歧只有一个：**产品方案 Section 7.4 对权限的强硬承诺和技术方案 Section 6.1 的实现现状之间有落差**，需要在对外沟通（如用户文档、MVP 介绍）中统一口径。其他问题都是补全，不是方向分歧。

## 10. 产品-技术对齐意见

> 本节是郭聪明读完阿构技术方案（docs/proposals/aion-app-store-tech.md）后的产品侧回应。
> 重点针对技术方案 Section 9 的四个待决策项给出产品判断，同时记录产品-技术不一致的地方。

### 10.1 针对技术方案 Section 9 四个待决策项的产品判断

---

**决策 1：动态协商是否需要人工确认节点？**

**产品判断：P1 阶段需要，P0 不涉及（因为 P0 不做动态协商）。**

到 P1 引入动态协商时，协商完成后必须有轻量的人工确认节点，理由：

- 协商是 Agent 自动决定"谁来改造自己"——这涉及运行时代码注入，用户不知情不可接受
- 确认形式不需要展示完整 schema（用户看不懂），展示"语音 App 将修改输出格式以适配笔记 App，是否允许？"即可
- 确认结果持久化，下次同一对 App 连接时不再询问（除非版本变化触发重协商）

不加确认节点的风险：用户的 App 被另一个 App 的 Agent 静默改造行为，这在安全性和信任感上都过不去。

---

**决策 2：协商失败的降级策略（超时后断开 vs 回退静态 schema）？**

**产品判断：回退静态 schema，不直接断开。**

理由：
- 直接断开对用户来说是"装了等于没装"，体验差
- 如果 App 在 `capabilities` 里声明了静态 schema（`inputSchema` / `outputSchema`），降级到静态连接至少让基本功能跑起来
- 如果连静态 schema 都没有声明，再断开，并明确告知用户"两个 App 无法自动对接，请联系开发者"

**超时时间产品建议**：30 秒。超过 30 秒的协商对话对用户来说等待感已经很糟糕，不值得继续。

---

**决策 3：本地 Store 的 App 来源——仅手动安装还是 P0 就支持内置 App 目录？**

**产品判断：P0 必须有至少 2 个 Aion 官方示例 App 内置，不能只有手动安装。**

理由：
- 纯手动安装（用户自己找 .aionapp 文件）让 P0 毫无发现性，开发者如何验证生态跑通？
- 内置 2 个官方示例 App（推荐：一个"GitHub Notifications 摘要 App"、一个"Slack 消息转发 App"）充当生态种子，同时作为规范的活文档
- 技术上复用现有 Hub bundled-index.json 机制，不需要额外基建，成本低

但不需要 P0 就做完整的远程发现——内置 2 个 + 支持从文件安装就够了。

---

**决策 4：Worker Thread vs Child Process 沙箱强度？**

**产品判断：V0 接受 Worker Thread，但有前提条件。**

前提条件（必须在 P0 实现，不能推迟）：

> 技术方案 Section 6.1 明确指出：L2 运行时 permission enforcement（`checkPermission()` 在 apiHandlers 里的实际执行）目前是**空缺**（代码注释标明是 P2）。

如果 L2 不补上，产品方案里"未声明权限则拒绝执行"就是空承诺——manifest 里没有声明 shell，但 App 代码调用了 `aion.git.run()`，没有 L2，这个调用会直接通过。

**产品立场**：L2 必须和 P0 一起交付，不能是 P2。否则产品对用户的安全承诺（"App 只能使用声明过的权限"）是谎言。

Worker Thread 隔离强度不够的问题（CPU/内存无法完全隔离），P0 可以接受，但产品文档里需要如实告知用户这个限制，不能过度承诺"完全沙箱隔离"。

### 10.2 产品-技术不对齐问题清单

| # | 问题 | 产品描述 | 技术现状 | 需要谁解决 |
|---|------|---------|---------|-----------|
| 1 | L2 权限执行空缺 | "未声明权限运行时拒绝" 是 P0 承诺 | 技术方案标记为 P2 | 架构组必须将 L2 提前到 P0 |
| 2 | 场景二描述与 P0 范围矛盾 | 场景二描述 AI 动态改造自身（语音 App 运行时注入适配代码） | 动态协商是 P1 | 产品方案场景二需加说明：该场景描述的是 P1 目标态，P0 用静态 schema 声明实现基础版本 |
| 3 | 信箱同步 RPC | 产品方案仅描述异步 Mailbox | 技术方案 `correlationId` 支持 request-reply 对，但未定义同步等待语义 | 明确：P0 不支持同步 RPC，`correlationId` 是追踪用，不是同步等待机制 |
| 4 | Agent 数量限制 | 产品方案写"每个 App 先允许 1 个主 Agent" | 技术方案 agentPool 默认限制 5 个/App | 对齐为：P0 限制 3 个 Agent/App（1 主 + 2 辅），超出报错不是静默失败 |
